# CLAUDE.md

This file provides guidance to Claude Code (claude.ai/code) when working with code in this repository.

## Commands

Solution file is `TaskTracker.slnx` (the new XML solution format). All `dotnet` commands accept it directly.

- Restore: `dotnet restore TaskTracker.slnx`
- Build: `dotnet build TaskTracker.slnx`
- Run the API (Development, http profile on http://localhost:5108): `dotnet run --project TaskTracker.Api --launch-profile http`
  - https profile (https://localhost:7062): `dotnet run --project TaskTracker.Api --launch-profile https`
- Run all tests: `dotnet test TaskTracker.slnx`
- Run a single test: `dotnet test --filter "FullyQualifiedName~<TestName>"`

When the API is running:
- Health: `/health`
- Ping (FastEndpoints smoke endpoint): `/ping`

In Development:
- Swagger JSON: `/swagger/v1/swagger.json`
- Scalar API reference UI: `/scalar/v1`

## Architecture

Clean architecture across six projects (four production + two test). Reference direction is enforced by project references — never reverse them:

```
Api ──► Application ──► Domain
  └───► Infrastructure ──► Application ──► Domain

Domain.UnitTests       ──► Domain
Api.IntegrationTests   ──► Api  (in-memory via WebApplicationFactory<Program>)
```

- `TaskTracker.Domain` — entities, value objects, domain rules. **No outward references.**
- `TaskTracker.Application` — use cases + the contracts (`IRepository<T>`, `ITaskService`, etc.) that Infrastructure must satisfy. Exposes `AddApplication()` extension on `IServiceCollection`.
- `TaskTracker.Infrastructure` — persistence, external integrations. Implements Application contracts. Exposes `AddInfrastructure(IConfiguration)`.
- `TaskTracker.Api` — composition root. Wires `AddApplication()` + `AddInfrastructure(...)`. Hosts the HTTP server.
- `TaskTracker.Domain.UnitTests` — fast unit tests on Domain rules.
- `TaskTracker.Api.IntegrationTests` — `WebApplicationFactory<Program>`-based in-memory tests against the real HTTP pipeline.

### API layer conventions

- **FastEndpoints** is the HTTP layer (not MVC controllers, not raw Minimal API). Auto-discovered via `AddFastEndpoints()` / `UseFastEndpoints()` in `Program.cs`. The endpoint API uses `Send.OkAsync(...)` etc. (FE v7), not the older `SendAsync(...)`.
- Endpoints live under `TaskTracker.Api/Features/<FeatureName>/...` (vertical-slice / feature-folder layout). See `Features/Diagnostics/PingEndpoint.cs` for the canonical pattern.
- OpenAPI is generated by `FastEndpoints.Swagger` (`o.ShortSchemaNames = true`). Scalar UI is mapped only in Development.
- `Program.cs` ends with `public partial class Program;` — required so `WebApplicationFactory<Program>` (in `TaskTracker.Api.IntegrationTests`) can see the entry point. Don't remove it.
- `TimeProvider.System` is registered as a singleton — inject `TimeProvider` instead of using `DateTime.UtcNow` in domain/application code so tests can swap in `FakeTimeProvider`.
- Global exception handling: `AddProblemDetails()` + `app.UseExceptionHandler()` — uncaught exceptions surface as RFC 7807 ProblemDetails. Don't add try/catch around endpoint handlers just to format errors.
- Middleware order matters: `UseExceptionHandler` → `UseStatusCodePages` → `UseHttpsRedirection` → `UseFastEndpoints` → `MapHealthChecks`. Don't reorder unless you know why.

### Build-wide settings

- `Directory.Build.props` centralizes `TargetFramework=net10.0`, `LangVersion=latest`, `Nullable=enable`, `ImplicitUsings=enable`, `TreatWarningsAsErrors=true`, `EnforceCodeStyleInBuild=true`, `AnalysisMode=All`. Individual `.csproj` files **must not duplicate these properties**.
- `Directory.Packages.props` — Central Package Management is on (`ManagePackageVersionsCentrally=true`) **plus** transitive pinning (`CentralPackageTransitivePinningEnabled=true`). Add new dependencies as `<PackageVersion ... />` here, then reference from `.csproj` with `<PackageReference Include="..." />` (no `Version` attribute).
- `global.json` pins SDK to `10.0.203` with `rollForward=latestFeature`.
- `.editorconfig` is detailed and authoritative — Allman braces, `var` for built-ins, no `this.` qualification, `_camelCase` private fields, `s_` static prefix, etc. Follow it rather than guessing.
- Test projects suppress `CA1707` via `<NoWarn>` to allow xUnit's `Method_Scenario_Result` naming with underscores.

### Adding a new feature (typical path)

1. **Domain** — entity + value objects + invariants.
2. **Application** — define repository/service interface, write use case (handler) that takes those interfaces.
3. **Infrastructure** — implement the interfaces; register the implementations in `AddInfrastructure(...)`.
4. **Api** — add endpoint class under `Features/<FeatureName>/`; FE auto-discovers it.
5. **Tests** — unit tests against Domain/Application; integration test against the endpoint via `WebApplicationFactory<Program>`.
